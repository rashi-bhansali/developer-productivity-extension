# ADR-006: Adapter/Registry Pattern for Multi-Language Syntax Highlighting

## Context and Problem Statement

DevInks initially supported only Python for code cells, with syntax highlighting and syntax checking logic implemented in a single monolithic `SyntaxHighlighter.js` file. As the need to support additional languages (JavaScript, C++) arose, it became clear that continuing to add language-specific logic into a single file would result in an increasingly large, tightly coupled, and hard-to-maintain module.

We needed an architecture that would allow new languages to be added independently, tested in isolation, and registered without modifying core highlighting logic — while keeping the interface consistent across all languages.

---

## Decision Drivers

1. **Extensibility** : Adding a new language should require creating one new file, not modifying existing ones.
2. **Separation of Concerns** : Language-specific logic (keywords, patterns, syntax checks) should live in its own module, not be mixed with orchestration logic.
3. **Testability** : Each language adapter should be unit-testable in complete isolation.
4. **Consistency** : All languages must expose the same interface so the rest of the application can treat them uniformly.
5. **Maintainability** : Removing or updating a language should have no side effects on other languages.

---

## Considered Options

### 1. **Monolithic SyntaxHighlighter with Language Conditionals (Rejected)**

* **Pros** :
* Simple to implement initially.
* All logic in one place.
* **Cons** :
* Grows unbounded as languages are added.
* Language logic is interleaved — a change to Python risks breaking JavaScript.
* Difficult to test individual languages without running the whole module.
* Violates the Open/Closed Principle — adding a language requires modifying the core file.

### 2. **Adapter/Registry Pattern (Chosen Strategy)**

* **Pros** :
* Each language is a self-contained adapter module with a consistent interface.
* A central registry maps language IDs to adapters — new languages are added by registering them, not by editing existing code.
* Adapters are independently unit-testable.
* The orchestrator (`SyntaxHighlighter.js`) is decoupled from language specifics entirely.
* **Cons** :
* Slightly more files and initial setup compared to a single-file approach.
* Requires discipline to keep the adapter interface consistent as languages evolve.

### 3. **Plugin System with Dynamic Imports (Partially Considered)**

* **Pros** :
* Languages could be loaded on demand, reducing initial bundle size.
* **Cons** :
* Adds async complexity to what is currently a synchronous highlighting pipeline.
* Unnecessary overhead for a small, fixed set of languages in an extension context.
* Dynamic imports are harder to test and debug in a Chrome extension environment.

---

## Decision

We adopted the adapter/registry pattern. Each language (Python, JavaScript, C++) is implemented as an independent ES module exposing a fixed interface:

```javascript
export default {
  id,           // string: unique language identifier e.g. 'python'
  displayName,  // string: shown in the language selector dropdown
  highlight,    // (code: string) => string: returns syntax-highlighted HTML
  checkSyntax,  // (code: string) => { message, line, column }[]: returns errors
};
```

A central registry (`registry.js`) imports all adapters and exposes two functions:

```javascript
getSupportedLanguages() // returns [{ id, displayName }] for the UI dropdown
getAdapter(languageId)  // returns the adapter or undefined
```

`SyntaxHighlighter.js` acts as the orchestrator — it calls `getAdapter()` and delegates all language-specific work to the returned adapter, with no knowledge of any individual language.

---

## Consequences

### Positive Consequences

* **Open for extension, closed for modification** : Adding a new language (e.g. TypeScript, Java) requires only creating a new adapter file and registering it — zero changes to existing code.
* **Isolated testing** : Each adapter has its own dedicated test file (`PythonAdapter.test.js`, `JavascriptAdapter.test.js`, `CppAdapter.test.js`) with no shared state.
* **Language persistence** : Because each adapter has a stable `id`, language selection can be persisted to IndexedDB and reliably restored across sessions.
* **Consistent UI** : `getSupportedLanguages()` drives the language selector dropdown dynamically — the UI never needs to know which languages exist.

### Negative Consequences

* **Interface discipline required** : If an adapter omits `checkSyntax` or returns the wrong shape, the orchestrator will fail at runtime. This is mitigated by the registry shape tests in `LanguageAdapterRegistry.test.js`.
* **More files** : Three languages = three adapter files + one registry file, compared to one monolithic file. Acceptable trade-off for the modularity gained.

---

## Trade-Offs

* **Simplicity vs. Extensibility** : A single file is simpler for one language. The adapter pattern pays off immediately at two languages and compounds with each addition.
* **Static vs. Dynamic Registration** : We chose static imports in the registry over a dynamic plugin system. This keeps the architecture synchronous, simple, and fully testable — appropriate for the current scope.

---

## Related Architectural Decisions

* **ADR-005 (IndexedDB)** : Language IDs from the registry are persisted to IndexedDB as part of each `NoteCell`, enabling language state to survive session restarts.
* **Testing Framework (ADR-004)** : The adapter pattern was designed with testability as a first-class concern - each adapter and the registry itself have dedicated unit test suites running under Jest.

---

## Notes and Future Considerations

* **Adding a new language** : Create a new file in `src/js/components/CodeEditor/languages/`, implement the four exports (`id`, `displayName`, `highlight`, `checkSyntax`), and add it to `registry.js`. No other files need to change.
* **Shared utilities** : Common logic such as `escapeHtml` and `commonSyntaxChecks` (bracket matching, comment-aware parsing) lives in `src/js/utils/syntaxUtils.js` and is imported by all adapters to avoid duplication.
* **Interface enforcement** : A future improvement could introduce a validation step in the registry to assert that every registered adapter conforms to the expected interface at load time, surfacing missing methods early.
